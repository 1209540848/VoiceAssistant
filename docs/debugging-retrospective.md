# 问题排查记录

## 问题是什么

1. 一句话输入不精准，识别结果容易碎、容易错。
2. 前端会重复显示同一句话，导致文本不断叠加，像“复读”一样。

## 怎么排查的

1. 先看用户实际输出，确认不是单纯的 UI 显示问题，而是转写结果本身就不稳定。
2. 顺着链路看代码，检查浏览器采集、后端转写、前端渲染三个环节。
3. 在 `server.js` 里发现，服务端把每次 `result-generated` 的 `sentence.text` 都直接追加到数组里，最后又 `join("")` 拼接，导致同一句话的中间结果被反复累加。
4. 在前端里发现，麦克风采集是默认配置，没有明确限制单声道、降噪、回声消除和自动增益，容易放大噪声。

## 怎么解决的

1. 后端改掉错误的拼接方式，不再机械累加中间结果，改成按句子状态收敛，最终只保留稳定文本。
2. 前端麦克风采集加上约束：`channelCount: 1`、`echoCancellation: true`、`noiseSuppression: true`、`autoGainControl: true`。
3. UI 侧去掉多余的“风格切换”和伪优化区，只保留录音、实时转写、诊断和复制，减少重复显示和干扰。
4. 桌面端迁移采集入口：Electron 环境下不再优先用浏览器页面采集，而是由 Electron 主进程拉起 Windows 系统麦克风采集脚本，直接输出 `16k / 16-bit / mono PCM`，再发给本地 ASR WebSocket。

## 为什么迁移麦克风采集

之前虽然看起来是桌面应用，但本质是 Electron 窗口加载网页，录音仍然发生在 `web/app.js` 里：

```text
renderer 页面 -> getUserMedia -> AudioContext -> JS 下采样 -> 本地 WebSocket
```

这条链路的问题是：

- 采样率由浏览器和设备决定，通常不是 16k
- JS 下采样质量有限
- 浏览器权限和设备策略会影响录音稳定性
- 桌面端无法明确掌控采集过程

迁移后链路变成：

```text
Electron 主进程 -> Windows 系统麦克风 -> 16k PCM -> 本地 ASR WebSocket -> renderer 显示结果
```

renderer 只负责按钮和显示，不再承担桌面端默认录音职责。

## 新问题：测试录音会少一段

### 问题是什么

测试时发现录音结果会少一小段，通常表现为：

- 开头几个字没进入识别
- 结尾半句话没进入最终结果

### 为什么会发生

这不是单纯的麦克风坏了，而是录音链路启动和停止都有边界损耗。

开始时链路是：

```text
点击开始
-> Electron 主进程启动系统采集脚本
-> Windows 打开默认麦克风
-> 本地 WebSocket 连接 ASR 服务
-> 后端连接 DashScope
-> DashScope 返回 task-started
-> 音频稳定进入识别
```

如果用户点击开始后马上说话，前面一小段音频可能还没有被稳定送进 ASR。

停止时当前链路偏硬：

```text
点击停止
-> 发送 stop
-> kill 采集进程
-> ASR 收尾
```

问题是采集进程 stdout 里可能还有尾部音频没 flush 出来，WebSocket 队列里也可能还有最后几帧没发完。此时立即停止，就会丢掉结尾一小段。

### 后续解决方向

1. 开始录音后，等系统采集和 ASR stream 都 ready，再提示用户可以说话。
2. 停止录音时不要立即 kill 采集进程，先保留 300-800ms 尾音缓冲。
3. 等最后几帧音频发送完成后，再发送 `stop / finish-task`。
4. UI 上区分“准备中”“可说话”“收尾中”，避免用户误以为一点击就能完整录入。

### 体验修正

显式展示“可以说话”“收尾中”会影响输入手感，用户不应该感知底层流程。

新的设计是：

```text
用户体验：按下就说，松开就等结果
内部实现：预缓存 + 热服务 + 尾音缓冲
```

具体方案：

1. **预缓存**
   - 用户点击或按快捷键后，立刻开始系统麦克风采集。
   - 如果 ASR 还没 ready，音频先进入本地 `preBuffer`。
   - ASR ready 后，把 `preBuffer` 按顺序补发给 ASR。
   - 限制最大预缓存时长，例如 2 秒，避免内存无限增长。

2. **热服务**
   - 应用启动后保持本地服务常驻。
   - 启动桌面壳时主动请求 `/api/health`，确认服务可用。
   - 快捷键唤起时不用从零启动服务。
   - 后续可继续做 ASR 连接预热，但要注意 DashScope 空闲连接策略。

3. **尾音缓冲**
   - 用户停止录音时，UI 可以立刻表现为停止。
   - 后台继续保留 500ms 左右尾音。
   - 最后几帧音频发送完成后，再发送 `stop / finish-task`。

4. **悬浮输入条**
   - 日常输入不使用大窗口。
   - 使用小尺寸置顶悬浮条，包含麦克风按钮、实时文本和复制入口。
   - 内部状态机不直接暴露给用户，只用图标、波形和轻文案反馈。

## 当前主要问题归类

### 1. 录音边界丢失

表现：

- 开始录音后马上说话，前几个字容易丢。
- 停止录音时太快，最后几个字容易丢。

影响：

- 输入句子变残缺。
- ASR 和后续理解只能基于残缺文本判断，容易造成错误理解。

优先级：

- 最高。这个问题属于链路问题，最可控，应该优先修。

### 2. 发音不标准导致专有词识别错误

表现：

- 专有词、人名、技术词、业务词容易被识别成相近读音。
- 例如“实时转写”可能被识别成“实时撰写”。

影响：

- 关键词错误会直接改变句子含义。
- 办公、技术、会议场景中影响较大。

解决方向：

- 增加热词。
- 增加用户词库和技术词表。
- 对高频错词做后处理纠错。

### 3. 中英混合识别不稳定

表现：

- `API`、`WebSocket`、`Electron`、`DashScope`、`PCM` 等词容易被识别成中文谐音或错误拼写。
- 英文缩写和中文句子混在一起时更容易错。

影响：

- 技术场景输入不稳定。
- 搜索、代码沟通、问题排查类输入容易失真。

解决方向：

- 增加中英混合热词表。
- 对常见技术词做固定替换。
- 后续用上下文纠错，把明显错词改回正确技术词。

## 当前判断

当前主要问题不是 ASR 完全不可用，而是：

```text
录音边界丢失导致上下文残缺
+ 发音不标准导致近音词错误
+ 专有词和中英混合识别难度高
= 最终文本容易误解或错词
```

下一步应优先解决录音边界，再处理热词和后处理纠错。

## 结论

问题核心不是单点故障，而是三处叠加：

- 音频采集不够稳
- 后端结果合并方式不对
- 前端展示把中间结果反复放大了

修复后，重复显示明显减少，转写链路也更接近真实可用状态。
