# 问题排查记录

## 问题是什么

1. 一句话输入不精准，识别结果容易碎、容易错。
2. 前端会重复显示同一句话，导致文本不断叠加，像“复读”一样。

## 怎么排查的

1. 先看用户实际输出，确认不是单纯的 UI 显示问题，而是转写结果本身就不稳定。
2. 顺着链路看代码，检查浏览器采集、后端转写、前端渲染三个环节。
3. 在 `server.js` 里发现，服务端把每次 `result-generated` 的 `sentence.text` 都直接追加到数组里，最后又 `join("")` 拼接，导致同一句话的中间结果被反复累加。
4. 在前端里发现，麦克风采集是默认配置，没有明确限制单声道、降噪、回声消除和自动增益，容易放大噪声。

## 怎么解决的

1. 后端改掉错误的拼接方式，不再机械累加中间结果，改成按句子状态收敛，最终只保留稳定文本。
2. 前端麦克风采集加上约束：`channelCount: 1`、`echoCancellation: true`、`noiseSuppression: true`、`autoGainControl: true`。
3. UI 侧去掉多余的“风格切换”和伪优化区，只保留录音、实时转写、诊断和复制，减少重复显示和干扰。
4. 桌面端迁移采集入口：Electron 环境下不再优先用浏览器页面采集，而是由 Electron 主进程拉起 Windows 系统麦克风采集脚本，直接输出 `16k / 16-bit / mono PCM`，再发给本地 ASR WebSocket。

## 为什么迁移麦克风采集

之前虽然看起来是桌面应用，但本质是 Electron 窗口加载网页，录音仍然发生在 `web/app.js` 里：

```text
renderer 页面 -> getUserMedia -> AudioContext -> JS 下采样 -> 本地 WebSocket
```

这条链路的问题是：

- 采样率由浏览器和设备决定，通常不是 16k
- JS 下采样质量有限
- 浏览器权限和设备策略会影响录音稳定性
- 桌面端无法明确掌控采集过程

迁移后链路变成：

```text
Electron 主进程 -> Windows 系统麦克风 -> 16k PCM -> 本地 ASR WebSocket -> renderer 显示结果
```

renderer 只负责按钮和显示，不再承担桌面端默认录音职责。

## 结论

问题核心不是单点故障，而是三处叠加：

- 音频采集不够稳
- 后端结果合并方式不对
- 前端展示把中间结果反复放大了

修复后，重复显示明显减少，转写链路也更接近真实可用状态。
