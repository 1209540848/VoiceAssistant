# 桌面端转型方案

## 1. 转型结论

项目主方向从 Web Demo 转向桌面端应用。

原因是本产品的核心价值不是“打开网页后录音”，而是用户在任意输入场景中都能快速唤起语音输入助手，说话后得到可直接使用的文本，并快速复制或粘贴到当前应用中。

桌面端更适合承载以下核心能力：

- 全局快捷键唤起。
- 系统托盘常驻。
- 悬浮输入条。
- 更稳定的麦克风权限与音频采集。
- 自动复制或粘贴到当前输入场景。
- 后续扩展到微信、飞书、文档、浏览器等桌面工作流。

## 2. 现有 Web 代码的定位

当前 Web 端不会废弃，而是作为桌面端的第一版 renderer 原型复用。

现有 `web/index.html`、`web/app.js`、`web/styles.css` 已经沉淀了以下能力：

- 主界面布局。
- 录音按钮和状态展示。
- 原始转写区。
- 优化结果区。
- 风格切换区。
- 复制、重新生成、清空等操作入口。
- DashScope ASR 接入和流式转写方向验证。
- 错误提示和 trace 信息展示。

因此桌面端第一阶段会直接复用现有 Web 布局，让 Electron 加载当前页面，先验证桌面壳、常驻和唤起能力。

## 3. Web 布局是否复用

结论：复用，但不是长期原样复用。

短期复用方式：

- 将当前 Web 页面作为 Electron renderer 的主窗口。
- 保留现有双栏结构：左侧原始转写，右侧优化结果。
- 保留现有录音按钮、状态栏、风格切换和操作按钮。
- 保留现有 CSS 作为桌面主窗口的基础视觉样式。

中期拆分方式：

- 主窗口：用于完整查看原文、优化结果、历史记录和设置。
- 悬浮输入条：用于高频语音输入，只保留录音按钮、实时转写、结果确认和复制/粘贴动作。

长期产品形态不应依赖完整 Web 页面。桌面端的主体验应该是轻量悬浮输入条，当前 Web 布局主要用于主窗口和开发调试。

## 4. 推荐技术路线

首版桌面端使用 Electron。

原因：

- 可以直接复用现有 HTML、CSS、JavaScript。
- 可以继续使用 Node.js 能力调用本地服务和 WebSocket。
- 支持全局快捷键、托盘、窗口控制、剪贴板和系统集成。
- 对当前项目迁移成本最低。

推荐结构：

```text
VoiceAssistant/
  desktop/
    main.js
    preload.js
  web/
    index.html
    app.js
    styles.css
  server.js
  package.json
```

## 5. 桌面端阶段规划

### 阶段一：Electron 壳

目标：让当前 Web MVP 在桌面应用中跑起来。

任务：

- 新增 Electron 主进程。
- Electron 启动时打开当前 `web/index.html`。
- 启动或内嵌现有 `server.js`。
- 保留 `.env.local` 配置方式。
- 打包成 Windows 可运行桌面应用。

验收：

- 双击应用即可打开。
- 不需要手动运行 `npm start`。
- 当前录音、转写和结果展示能力在桌面窗口中可用。

### 阶段二：系统常驻

目标：让应用具备桌面工具属性。

任务：

- 增加系统托盘。
- 关闭窗口时隐藏到托盘。
- 托盘菜单提供显示窗口和退出。
- 后续支持开机自启动。

验收：

- 应用可以常驻后台。
- 用户可以从托盘重新打开。
- 退出动作明确，不误关服务。

### 阶段三：全局快捷键

目标：从任意应用中快速唤起语音输入。

任务：

- 注册全局快捷键，例如 `Ctrl+Shift+Space`。
- 快捷键显示或隐藏语音输入窗口。
- 窗口置顶。

验收：

- 用户在微信、飞书、浏览器、文档中都能唤起。
- 不需要先切换到 VoiceAssistant。

### 阶段四：悬浮输入条

目标：把主页面能力拆成更适合高频使用的小窗口。

任务：

- 从当前 Web 布局中抽出精简模式。
- 只保留录音按钮、实时转写、最终结果和复制/粘贴动作。
- 支持置顶、拖动、自动收起。

验收：

- 不遮挡用户主要工作界面。
- 使用路径足够短：唤起 -> 说话 -> 得到文本 -> 复制/粘贴。

### 阶段五：文本输出到当前应用

目标：让结果真正进入用户当前工作流。

任务：

- 先支持一键复制到剪贴板。
- 再支持自动复制后模拟粘贴。
- 后续根据应用场景增加确认机制。

验收：

- 用户能快速把文本送进当前输入框。
- 自动粘贴失败时有明确提示，不丢失文本。

## 6. 需要保留和改造的代码

直接保留：

- `server.js` 中的 DashScope 接入和 WebSocket 转写逻辑。
- `web/index.html` 的主页面结构。
- `web/styles.css` 的基础布局和视觉样式。
- `.env.local` 的本地配置方式。

需要改造：

- `web/app.js`：逐步拆分录音、转写、UI 状态、复制粘贴逻辑。
- `web/index.html`：拆成主窗口和悬浮窗两种渲染模式。
- `server.js`：后续考虑内嵌进 Electron 主进程或作为本地子进程启动。
- `package.json`：增加 Electron 启动和打包脚本。

## 7. 当前决策

当前 Web 端继续作为能力验证和桌面主窗口原型存在。

下一步不继续把 Web 页面打磨成独立 Web 产品，而是先完成 Electron 桌面壳，让现有页面在桌面环境中运行起来，再逐步加入托盘、全局快捷键、悬浮输入条和复制/粘贴能力。
