# 语音输入边界测试文本

用于测试语音输入助手在不同输入边界下的表现。每条测试时建议记录：

- 是否漏开头
- 是否漏结尾
- 是否重复显示
- 是否识别错关键词
- 是否自动断句合理

观察点：短句是否快速返回，是否误判为空。

1. 好。
2. 可以。
3. 不行。
4. 明白。
5. 稍等。

## 6. 专有名词和技术词

观察点：产品名、模型名、英文缩写是否稳定。

1. 我们现在使用的是 DashScope 的 fun-asr-realtime 模型。

3. 前端通过 preload 暴露 startRecording 和 stopRecording。
4. 服务器通过 WebSocket 把音频帧转发给 ASR 服务。
5. 这次要验证 PCM 十六 k 单声道音频是否稳定。

## 9. 长句

观察点：长句是否断句，是否漏中间内容。

1. 我们现在遇到的问题是用户点击开始录音以后，如果马上开始说话，前面几个字可能还没有进入稳定的识别链路，所以最终结果看起来像是少录了一段。
2. 桌面端迁移以后，录音不再由浏览器页面采集，而是由 Electron 主进程启动 Windows 系统麦克风采集脚本，再把十六 k 的 PCM 音频送给本地 WebSocket 服务。
3. 后端之前的问题是把每一次中间识别结果都当成新的文本追加起来，最后直接拼接成最终结果，所以同一句话会在前端看起来重复很多遍。

## 10. 停止边界

观察点：点击停止前后的尾音是否丢失。

1. 请在我说完这句话以后，等一秒钟再点停止。
2. 请在我说到这里马上点停止，观察最后几个字是否丢失。
3. 这句话的最后几个字非常重要，不应该被截断。
4. 我现在说完最后一个词，测试尾音缓冲。
5. 停止录音之前不要太快，否则可能丢掉结尾。

## 11. 开始边界

观察点：点击开始后立刻说话是否丢开头。

1. 一点击开始马上说这句话。
2. 开始以后等一秒再说这句话。
3. 第一秒说的内容可能会丢失。
4. 等状态显示收音中以后再开始说话。
5. 这条用来测试开头是否完整。

## 15. 建议测试顺序

1. 先测极短句和开始边界，确认是否丢开头。
2. 再测停止边界，确认是否丢结尾。
3. 再测数字、专有名词和中英混合。
4. 最后测长句和会议记录。
