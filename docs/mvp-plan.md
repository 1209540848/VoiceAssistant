# AI 语音输入助手 MVP 开发计划

## 1. MVP 目标

首版 MVP 聚焦 Web 端语音输入助手，先跑通最核心的输入闭环：

> 用户说话 -> 实时出字 -> AI 优化 -> 用户确认使用

MVP 不追求完整平台能力，也不急于做跨应用语音操作。第一阶段的重点是验证用户是否愿意在聊天、办公、搜索、记录和写作场景中，用语音生成可直接使用的文本。

## 2. 首版平台选择

当前平台策略调整为：桌面端优先，Web 端作为桌面端 renderer 原型和能力验证环境保留。

此前选择 Web Demo 的原因：

- 开发和调试成本低。
- 浏览器已支持麦克风授权和基础录音能力。
- 易于快速验证界面、转写、AI 优化和复制使用流程。
- 后续可复用核心服务能力到移动端和桌面端。

新的判断是：本产品的核心使用场景更接近桌面输入助手，而不是独立网页工具。用户需要在微信、飞书、浏览器、文档、邮件等任意输入场景中通过全局快捷键唤起语音输入，并把结果快速复制或粘贴到当前应用。

因此后续主线改为桌面端：

- 使用 Electron 承载现有 Web 页面。
- 短期复用当前 Web 布局作为桌面主窗口。
- 中期拆分出更轻量的悬浮输入条。
- 长期围绕托盘、全局快捷键、自动复制/粘贴和桌面工作流优化。

桌面端详细方案见：[desktop-transition-plan.md](desktop-transition-plan.md)。

## 3. MVP 范围

### 3.1 必做能力

- 麦克风授权和录音。
- 语音输入按钮。
- 实时或准实时转写结果展示。
- 原始转写文本展示。
- AI 优化文本展示。
- 文本风格切换：原文、正式、简洁、礼貌、总结。
- 复制结果。
- 重新生成。
- 清空内容。
- 基础错误提示：麦克风未授权、网络异常、识别失败、AI 优化失败。

### 3.2 可选能力

- 历史记录。
- 基础用户词库。
- 低置信度词高亮。
- 分段长文本处理。
- 简单语音编辑命令，例如“删除上一句”“整理成三点”。

### 3.3 暂不做

- 移动端原生 App。
- 桌面端全局悬浮窗。
- 跨应用自动操作。
- 企业组织词库。
- 私有化部署。
- 完整方言覆盖。
- 完整多端同步。

## 4. 核心页面结构

MVP 只需要一个主页面。

### 4.1 页面区域

- 顶部状态区：展示当前状态，例如待输入、录音中、识别中、优化中、已完成。
- 语音输入区：主语音按钮，支持点击开始、再次点击结束。
- 原始转写区：展示 ASR 返回的原始文本。
- AI 优化区：展示优化后的文本。
- 风格选择区：原文、正式、简洁、礼貌、总结。
- 操作区：复制、重新生成、清空。
- 错误提示区：展示麦克风、网络、识别、优化相关错误。

### 4.2 推荐交互

1. 用户点击语音按钮。
2. 浏览器请求麦克风授权。
3. 授权成功后进入录音状态。
4. 系统展示实时或准实时转写。
5. 用户停止录音。
6. 系统生成最终原始转写。
7. AI 根据当前风格生成优化结果。
8. 用户复制原文或优化结果。

## 5. 技术方案建议

### 5.1 前端

建议使用 React 或 Next.js。

首版如果追求速度，可以使用 Vite + React；如果后续要做接口路由、登录和部署，可以直接使用 Next.js。

核心前端模块：

- `Recorder`：负责麦克风授权、录音开始、录音结束、音频数据管理。
- `TranscriptionPanel`：展示原始转写。
- `PolishPanel`：展示 AI 优化结果。
- `StyleSelector`：选择文本风格。
- `StatusBar`：展示录音、识别、优化和错误状态。

### 5.2 后端

首版建议提供两个接口：

| 接口 | 方法 | 作用 |
| --- | --- | --- |
| `/api/transcribe` | `POST` | 接收音频，返回原始转写文本。 |
| `/api/polish` | `POST` | 接收原始文本和风格，返回优化文本。 |

后续如果要做真正流式识别，可以增加 WebSocket 或 SSE：

| 接口 | 协议 | 作用 |
| --- | --- | --- |
| `/api/transcribe/stream` | WebSocket 或 SSE | 返回中间识别结果和最终结果。 |

### 5.3 数据结构草案

转写请求：

```json
{
  "audio": "binary audio file",
  "language": "zh-CN",
  "scene": "chat"
}
```

转写响应：

```json
{
  "text": "明天下午三点我们开个会讨论一下这个方案",
  "confidence": 0.92,
  "segments": [
    {
      "text": "明天下午三点",
      "start": 0,
      "end": 1.4,
      "confidence": 0.94
    }
  ]
}
```

优化请求：

```json
{
  "text": "明天下午三点我们开个会讨论一下这个方案",
  "style": "formal",
  "scene": "work"
}
```

优化响应：

```json
{
  "text": "我们明天下午三点召开会议，讨论该方案。",
  "style": "formal",
  "changed": true
}
```

## 6. 文本风格定义

| 风格 | 说明 | 示例 |
| --- | --- | --- |
| 原文 | 保留原始转写，仅做必要标点。 | 明天下午三点我们开个会，讨论一下这个方案。 |
| 正式 | 适合办公、邮件、通知。 | 我们将于明天下午三点召开会议，讨论该方案。 |
| 简洁 | 去掉冗余表达，适合聊天和搜索。 | 明天下午三点开会讨论方案。 |
| 礼貌 | 语气更柔和，适合沟通协作。 | 方便的话，我们明天下午三点开会讨论一下这个方案。 |
| 总结 | 提炼核心信息。 | 会议时间：明天下午三点；主题：讨论方案。 |

## 7. 开发任务拆分

### 7.1 第一轮：静态可用界面

- 搭建 Web 项目。
- 实现主页面布局。
- 实现语音按钮、状态区、原文区、优化区、风格选择和操作按钮。
- 使用假数据模拟转写和优化流程。
- 验证页面在桌面和移动宽度下可用。

验收标准：

- 用户能点击按钮进入模拟录音状态。
- 用户停止后能看到模拟原文和模拟优化结果。
- 风格切换能影响模拟输出。
- 复制、重新生成、清空按钮可用。

### 7.2 第二轮：接入真实录音

- 接入浏览器麦克风授权。
- 实现录音开始和停止。
- 将音频转为可上传格式。
- 增加麦克风未授权提示。
- 增加录音中状态反馈。

验收标准：

- 用户首次使用时浏览器会请求麦克风权限。
- 授权成功后可以录音。
- 拒绝授权时有明确提示。
- 停止录音后能获得音频数据。

### 7.3 第三轮：接入转写接口

- 实现 `/api/transcribe`。
- 前端上传音频。
- 后端调用 ASR 服务或临时模拟 ASR。
- 返回原始转写结果。
- 增加转写失败提示。

验收标准：

- 用户录音后能得到真实或模拟转写文本。
- 网络异常或接口失败时不清空已录音状态。
- 用户可以重新录制。

### 7.4 第四轮：接入 AI 优化接口

- 实现 `/api/polish`。
- 根据风格生成优化文本。
- 支持重新生成。
- 增加优化失败降级策略。

验收标准：

- 用户得到原始转写后，系统能生成优化结果。
- 切换风格后可重新生成。
- AI 优化失败时保留原文。

### 7.5 第五轮：体验打磨

- 优化移动端布局。
- 增加加载状态和错误状态。
- 增加基础历史记录。
- 增加文案和空状态。
- 做一次完整用户路径测试。

验收标准：

- 用户从进入页面到复制结果不需要学习复杂操作。
- 常见失败场景都有明确提示。
- 页面在桌面和移动端不出现明显布局问题。

## 8. 关键体验验收

MVP 可以用以下问题判断是否达标：

- 用户是否能在 1 次点击内开始语音输入？
- 用户是否能清楚知道当前是在录音、识别还是优化？
- 用户是否能看到原始转写和 AI 优化结果？
- AI 优化是否保留原意？
- 结果是否可以被一键复制或使用？
- 失败时是否保留用户已经输入的内容？
- 短句场景是否足够快，不打断表达节奏？

## 9. 后续演进

MVP 跑通后，再按价值和风险逐步扩展：

1. 增加流式转写，做到边说边出字。
2. 增加用户词库，提升人名、公司名、产品名和专业词准确率。
3. 增加聊天、邮件、会议、报告等场景模板。
4. 增加移动端适配或原生端能力。
5. 增加桌面端快捷键和悬浮窗。
6. 增加语音操作能力，例如打开应用、搜索内容、发送消息和创建日程。

## 10. 当前建议

下一步建议直接进入第一轮开发：先搭建 Web 项目并实现静态可用界面。即使转写和 AI 都先用假数据，也能快速验证核心交互是否顺手，再逐步替换成真实接口。
